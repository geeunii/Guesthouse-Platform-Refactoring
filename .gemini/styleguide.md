# 팀 코드 리뷰 가이드 (Gemini)

## Output Language (중요)
- **리뷰/요약/코멘트/섹션 제목까지 전부 한국어로 작성한다.**
- **Write ALL outputs in Korean (ko).**
- 영어는 아래 경우에만 허용한다:
  - 코드 식별자(변수/함수/클래스명), 라이브러리/프레임워크명
  - 에러 메시지/로그/스택트레이스 원문 인용
  - 파일 경로/명령어/설정 키 등 원문 표기
- 리뷰 본문에 영어 문장/단락을 작성하지 않는다.

---

## 리뷰 출력 형식
- 코멘트는 아래 형식을 지킨다:
  1) [Severity: HIGH/MEDIUM/LOW] 한 줄 요약
  2) 문제 설명 (왜 문제인지)
  3) 영향/리스크 (버그/보안/성능/유지보수)
  4) 개선안 (가능하면 코드 예시)
- 문제가 없으면 마지막에 **LGTM** 을 명시한다.

## 우선순위 기준
- HIGH: 보안 취약점, 인증/인가 누락, 데이터 손상/장애 가능성, 크래시
- MEDIUM: 엣지케이스 누락, 리소스 누수 가능성, 성능 저하(반복 렌더/쿼리), 유지보수성 크게 저하
- LOW: 컨벤션/가독성/사소한 개선

---

# React 컴포넌트 규칙
- 함수형 컴포넌트만 사용
- (JS) PropTypes 반드시 정의 / (TS) Props 타입 엄격히 정의
- 컴포넌트명 PascalCase
- interactive 요소를 div로 만들지 말고 button/a 등 semantic 요소 사용

# 성능 최적화
- 불필요한 리렌더링 원인(의존성 배열, inline 함수/객체 생성)을 우선 지적
- useMemo/useCallback은 “근거”가 있을 때만 권장 (남발 금지)
- 큰 리스트는 lazy loading/virtualization 고려

# 접근성 및 SEO
- 의미있는 alt 필수 (장식용 이미지는 alt="")
- semantic HTML 태그 사용(header/nav/main/section/button 등)
- ARIA는 필요할 때만 정확히 사용(잘못된 aria-*는 지적)

# (선택) 백엔드도 같이 보는 경우
- 입력 검증/권한 체크(인증/인가) 누락 여부
- 예외 처리/에러 메시지에 민감정보 포함 금지
- 불필요한 반복 쿼리, 페이징/인덱스 고려
