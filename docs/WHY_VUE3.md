# 왜 Vue3를 선택했는가?

> 이 문서는 프로젝트에서 Vue3를 프론트엔드 프레임워크로 선택한 이유와 기술적 배경을 설명합니다.

---

## 1. Vue3 선택 이유

### 1.1 학습 곡선이 완만하고 생산성이 높음
- Vue3는 React나 Angular에 비해 진입 장벽이 낮아 **팀원 모두가 빠르게 적응**할 수 있습니다.
- HTML 기반의 직관적인 템플릿 문법으로 기존 웹 개발 경험만 있으면 바로 시작 가능합니다.

### 1.2 Composition API 지원
- Vue3의 `Composition API`는 **로직 재사용과 코드 구조화**에 탁월합니다.
- 복잡한 검색 로직, 필터링, 상태 관리 등을 관심사별로 깔끔하게 분리할 수 있습니다.

### 1.3 Pinia를 통한 간결한 상태 관리
- `Pinia`는 Vue3의 공식 상태 관리 라이브러리입니다.
- 기존 Vuex보다 **더 간단하고 TypeScript 친화적**입니다.

### 1.4 Vite와의 완벽한 통합
- `Vite`는 Vue 팀이 만든 빌드 도구로 **빠른 HMR(Hot Module Replacement)**과 **최적화된 빌드**를 제공합니다.
- 개발 중 변경 사항이 **즉시 반영**되어 개발 생산성이 높습니다.

### 1.5 Spring Boot 백엔드와의 궁합
- RESTful API 기반 프로젝트에서 Vue3는 **백엔드와 완전히 분리된 SPA 개발**에 적합합니다.
- 프론트엔드/백엔드 역할 분담이 명확해집니다.

### 1.6 커뮤니티와 생태계
- 한국 개발자 커뮤니티에서 Vue 사용률이 높아 **자료와 레퍼런스**를 찾기 쉽습니다.
- `vue-router`, `chart.js`, `vue-chartjs` 등 필요한 라이브러리가 잘 갖춰져 있습니다.

---

## 2. Vue3 vs React 비교

| 항목 | Vue3 | React |
|------|------|-------|
| **학습 곡선** | 완만함 (HTML 기반 템플릿) | 상대적으로 가파름 (JSX, 함수형 패러다임) |
| **문법** | HTML 템플릿 + 디렉티브 (v-if, v-for) | JSX (JavaScript 안에 HTML) |
| **상태 관리** | Pinia (공식, 간단) | Redux/Zustand/Recoil 등 (선택지 많음) |
| **양방향 바인딩** | v-model 기본 지원 ✅ | 직접 구현 필요 (onChange + setState) |
| **번들 크기** | 더 작음 (~33KB gzipped) | 더 큼 (~42KB gzipped) |
| **TypeScript 지원** | 좋음 (Vue3에서 대폭 개선) | 매우 좋음 |
| **커뮤니티 규모** | 아시아권에서 인기 (한국, 중국) | 전세계적으로 대규모 |

### 프로젝트에서 Vue3가 적합했던 이유
- **v-model 양방향 바인딩**: 검색 필터처럼 폼 요소가 많은 UI에서 코드량을 크게 줄여줍니다.
- **공식 도구 통합**: vue-router, Pinia, Vite가 모두 공식 지원되어 설정이 간편합니다.

## 3. Options API vs Composition API

### 3.1 Options API (Vue2 스타일)
- 데이터, 메서드, computed, watch가 **각각 분리**되어 정의됩니다.
- 컴포넌트가 커지면 관련 로직이 여러 곳에 흩어져 **가독성이 떨어집니다**.
- 로직 재사용이 어렵습니다 (mixins는 충돌 위험 있음).

### 3.2 Composition API (Vue3 스타일)
- 관련 상태와 로직을 **한 곳에 모아** 관리할 수 있습니다.
- **기능별 그룹핑**: 가격 로직, 게스트 로직, 테마 로직을 각각 묶어서 관리합니다.
- **로직 재사용**: Composable 함수로 추출하여 여러 컴포넌트에서 재사용 가능합니다.
- **TypeScript 친화적**: 타입 추론이 자연스럽습니다.
- **테스트 용이**: 순수 함수로 분리하여 단위 테스트가 쉽습니다.

### 왜 Composition API를 선택했는가?
- FilterModal, SearchStore 등 복잡한 컴포넌트에서 **관심사별 코드 분리**가 필요했습니다.
- Pinia도 Composition API 스타일을 사용하여 **일관성**을 유지할 수 있었습니다.

## 4. SPA에서 Vue vs React: 어떤 게 더 좋은가?

### 결론: 둘 다 SPA에 적합하며, "더 좋다"고 단정할 수 없음

| 항목 | Vue3 | React | 우위 |
|------|------|-------|------|
| **라우팅** | vue-router (공식) | react-router (서드파티) | Vue ✅ |
| **상태 관리** | Pinia (공식) | Redux/Zustand 등 | 동등 |
| **번들 크기** | ~33KB | ~42KB | Vue ✅ |
| **SSR/SSG** | Nuxt | Next.js | React ✅ (더 성숙) |
| **모바일 앱** | 제한적 | React Native | React ✅ |
| **대규모 생태계** | 양호 | 매우 풍부 | React ✅ |
| **기업 채용** | 국내 중견/스타트업 | 글로벌 대기업 | React ✅ |
| **학습 곡선** | 완만함 | 상대적으로 가파름 | Vue ✅ |
| **폼/입력 처리** | v-model (간편) | 수동 바인딩 | Vue ✅ |

### Vue가 SPA에서 유리한 상황
1. **폼 중심 애플리케이션**: 검색/필터 UI가 많은 경우
2. **빠른 프로토타이핑 & MVP**: 설정이 간단하고 팀원 온보딩이 빠름
3. **중소규모 프로젝트**: 오버엔지니어링 없이 깔끔한 구조 유지 가능

### React가 SPA에서 유리한 상황
1. **대규모 엔터프라이즈 앱**: 더 큰 생태계와 라이브러리 선택지
2. **크로스 플랫폼**: React Native로 모바일 앱까지 확장 가능
3. **SEO가 중요한 서비스**: Next.js의 SSR/SSG가 더 성숙
4. **취업/이직 고려**: 글로벌 채용 시장에서 React 수요가 더 높음

## 5. 면접/발표용 답변 예시

### 짧은 버전 (1분)
> "저희 팀은 Vue3를 선택했습니다. 그 이유는 첫째, **학습 곡선이 완만**해서 팀원 모두가 빠르게 적응할 수 있었고, 둘째, **Composition API**를 활용해 검색/필터링 같은 복잡한 로직을 재사용 가능한 형태로 구조화할 수 있었습니다. 셋째, **Vite**와의 통합으로 개발 속도가 빨랐고, 넷째, **Pinia**로 상태 관리를 간결하게 처리했습니다. 마지막으로 Spring Boot REST API와 **프론트/백 분리 구조**에 Vue가 적합했습니다."

### 후속 질문 대비

**Q: React 대신 Vue를 선택한 구체적인 이유는?**
> "React도 좋은 선택이지만, 저희 프로젝트에서는 Vue가 더 적합했습니다. 첫째, 팀원들의 기존 경험이 Vue에 더 익숙했고, 둘째, 검색 필터처럼 폼 요소가 많은 UI에서 v-model이 코드량을 크게 줄여줬습니다. 셋째, Spring Boot 백엔드와 분리된 SPA 구조에서 Vue가 충분했습니다."

**Q: Options API 대신 Composition API를 쓴 이유는?**
> "Options API는 데이터, 메서드, computed가 분리되어 컴포넌트가 커지면 관련 로직을 찾기 어렵습니다. Composition API는 기능별로 코드를 묶을 수 있어 FilterModal처럼 복잡한 컴포넌트에서 유지보수가 쉬웠습니다. 또한 Pinia와의 일관성도 있습니다."

**Q: Vue3의 단점은 없었나요?**
> "React에 비해 생태계가 작아 일부 라이브러리 선택지가 제한적이었지만, 저희 프로젝트에 필요한 vue-router, pinia, chart.js 등은 모두 잘 지원되어 문제 없었습니다."

**Q: SPA에서 Vue가 React보다 더 좋은가요?**
> "둘 다 SPA 구축에 완벽하게 적합합니다. 저희 프로젝트에서 Vue를 선택한 이유는 검색/필터 UI가 핵심이었는데 v-model로 폼 처리가 간편했고, 프로젝트 규모가 중규모였기 때문에 Vue의 간결함이 적합했습니다. 다만 React Native가 필요하거나 글로벌 서비스 확장이 목표라면 React도 좋은 선택입니다."

## 6. 핵심 정리

| 질문 | 답변 |
|------|------|
| **왜 Vue3를 선택했는가?** | 학습 곡선, v-model, Composition API, Vite/Pinia 통합, 팀 생산성 |
| **React 대신 Vue인 이유는?** | 폼 중심 UI에서 v-model 생산성, 공식 도구 통합, 팀 경험 |
| **Options API vs Composition API?** | Composition API로 관심사별 코드 분리, 로직 재사용, Pinia 일관성 |
| **Vue가 React보다 SPA에서 더 좋은가?** | 상황에 따라 다름. 우리 프로젝트에는 Vue가 적합했음 |
